PROBLEM & SOLUTION

超长视频生成（突破 15 秒上限）

R2V 链式拼接 + 4 层一致性锚定，60s 实战 demo 已跑通，已在 Studio 工坊上线

致命
● 实测短板

HappyHorse 单次生成上限 15 秒，长素材（30s / 60s / 90s 投流视频）无法直接产出，prompt 上约束失败。

最致命的是无法制作视频延长，只能最多 15s 视频，无法满足买量侧素材生成。
— 买量侧测试结论
● 我们的应对：R2V 链式生成（4 层一致性锚定）

把多个 15s 段串行拼接，全部段用 R2V（不再纯 I2V），通过 4 层锚定解决"角色漂移"和"尾帧无内容"两个核心痛点：

  1. 任务定义全局注入：顶部 textarea 写主角与场景描述，自动 prepend 到每段 prompt 开头
  2. 锚点参考图全段共享：从源 job 继承的角色/场景参考图，每段都喂进去，人物身份不再依赖单帧
  3. 段 2+ 切到 R2V 模式：每段 refs = 锚点 refs + 上段衔接帧。R2V 同时锚定"我是谁（锚点）"与"我刚才在做什么（衔接帧）"
  4. 尾帧 3 帧候选：本段完成后提取最后 1.5s 内 3 帧，UI 缩略让用户挑选（默认末帧；空镜/转场时可改选含主体的早帧）
  5. 循环重复，按段拼接
项目里已实现
Studio 工坊 · 「链式」按钮支持：任务定义输入框、锚点参考图多选、段 2+ R2V 模式、3 帧候选选择器（Studio 入口）。每段独立可重试、错误隔离、实时进度展示。
V1（旧 I2V） → V2（R2V + 锚点）核心差异
# V1：段 2 = i2v(上段尾帧)
       ↑ 只看 1 帧 → 3 段后角色漂移

# V2：段 2 = r2v(reference_urls: [锚点角色图, 锚点场景图, 用户选的尾帧])
       ↑ 身份 + 环境 + 衔接 三层锚定
HH 单段 10 秒极限范例：公园女子动作链 · 单段塞 5 个动作节点。下方是用 V2 链式真实跑出的 60 秒 demo（4 段 × 15s 拼接）。
CASE STUDY

60 秒超长视频实战记录

9:16 竖屏 · 5+4×15s · 共 5 次模型调用

主题：韩系治愈日常 vlog ——25 岁清秀女生的一日四时刻。构图策略：以 3/4 侧脸、手部、背影为主，规避正脸大特写——HH 长视频对脸部高频细节容易漂移，主动用构图把视觉焦点分散到运镜与场景上。

生产链路（5 段调用 + 拼接）
# Step 0  T2V 5s 种子段 → ffmpeg 提中段帧 → 上传 OSS（拿锚点）
# Step 1  R2V 15s · refs=[锚点]            → 卧室晨起
# Step 2  R2V 15s · refs=[锚点 + 段1尾帧]  → 衣帽间挑衣
# Step 3  R2V 15s · refs=[锚点 + 段2尾帧]  → 吧台冲咖啡
# Step 4  R2V 15s · refs=[锚点 + 段3尾帧]  → 玄关出门
# Step 5  ffmpeg concat -c copy 拼成 60s   → 无损拼接
关键技术决策
为什么段 1+ 全用 R2V，不用 I2V？I2V 只看 1 帧，4 段后主角脸明显漂移（见下方 V1 对照实验）；R2V 同时看锚点 + 衔接帧，人物身份持续锁定
为什么 4 段而不是 6 段 10s？段数少 → 漂移机会少；15s 是 HH 单段上限，4 段刚好 60s
为什么避开正脸大特写？HH 在长视频里对脸部高频细节（眼睛、嘴角）容易漂移；改用 3/4 侧脸 + 手部 + 背影构图，把视觉焦点分散到运镜与场景
拼接为什么 -c copy？不重编码 → 零质量损失；4 段同分辨率/帧率 9:16 720P，concat 协议直接拼
最终产出：60 秒 / 31.6 MB / 9:16 720P。4 段 × 15s 拼接，韩系治愈一日 vlog 叙事。已知限制：段 3、段 4 的场景切换（厨房、玄关）未完全执行到位，R2V 模式倾向于延续锚点环境，需在 prompt 中进一步强化场景切换指令或单独重跑。
ABLATION

V1 对照实验 · I2V 链式（同 prompt、同 anchor、无锚点参考）

仅用于对比

为验证 V1 → V2 的选型依据，用同一份 anchor.jpg + 完全一致的任务定义和段 prompt，跑了一版纯 I2V 链式 60s（4 段，每段起头帧 = 上段尾帧）。结果正向印证 R2V 选型：I2V 场景执行更听 prompt（咖啡杯、风衣都出现了），但 4 段后主角脸明显漂移成另一个人；R2V 因锚点参考压制场景切换换来身份稳定。两者是一对此消彼长的工程选择。

I2V 60s 对照视频：60 秒 / 34.9 MB / 9:16 720P，5 次模型调用省成 4 次（无 T2V 种子段）。
R2V（方案 A · 已选）赢点 主角身份 4 段一致；服装、发型、肤色无漂移；适合人物 IP 类、形象敏感场景
R2V 痛点 段 3、4 的场景被锚点拽住，"切到厨房/玄关"指令执行不到位
I2V（V1 对照）赢点 场景执行强：咖啡杯、风衣、玄关镜都按 prompt 出现；段间动作更丰富
I2V 痛点 4 段后主角脸明显漂移；适合不依赖主角身份的场景叙事（产品演示/旅拍空镜）

方案 B · Storyboard-as-Reference

待验证

GPT Image 2 一句话生成 12 格分镜图（单图自带 4 个镜头描述）；HH 在一段 15s 视频里按 storyboard 顺序演绎 4 个镜头。把多张参考图压缩成一张多格图作为内嵌一致性源，每 15s 段塞 4 镜头。横切 storyboard 3 行 → 3 段 = 45s；GPT 续写第二页 → 90s。

Prompt 模板
Generate a scene using the shots in the uploaded film storyboard.

First 4 seconds:  <shot 1 描述>
4–8 seconds:      <shot 2 描述>
8–12 seconds:     <shot 3 描述>
12–15 seconds:    <shot 4 描述>

no music, no subtitles
vs 方案 A（R2V 链式 · 已上线）
  • 调用次数：5 次 → 3-6 次（每段塞 4 镜头）
  • 一致性源：OSS 锚点参考图 → 单张多格 storyboard
  • 段间衔接：尾帧上传 → storyboard 顺序自带节奏
  • 图像端依赖：无 → GPT Image 2
未验证的风险点
  • HH 是否真的能按时间顺序"读"横向 storyboard
  • 15s 内塞 4 镜头（每镜 3.75s），节奏控制能否到位
  • GPT Image 2 跨页续写的一致性如何
  • "no music / no subtitles" 指令服从度
来源：Source #1 · ⭐ 技巧 1：Storyboard-as-Reference。下一步：跑通最小验证 demo（1 段 15s 测 4 镜头节奏 + 跨段一致性测试），决定是否升级为生产管线。

方案 C · 多 anchor 切换（解决 R2V 场景被锁问题）

待验证

从 60s 实战记录里看到的 R2V 已知限制：段 3、段 4 的场景切换执行不到位。原因是单一 anchor 把环境也锁住了。方案 C：每段配独立 anchor——段 1 用「卧室晨起」anchor，段 2 切到「衣帽间」anchor，段 3 切到「咖啡馆吧台」anchor，段 4 切到「玄关」anchor。主体一致性靠 character 锁定，环境一致性靠 anchor 切换。

vs 方案 A（单 anchor R2V）
  • 主体身份：A 单 anchor 锁住主角 + 环境；C 多 anchor 只锁主角，环境跟随 anchor 变
  • 场景叙事：A 段 3+4 场景偏离 prompt；C 每段都按 anchor 锁场景
  • 调用成本：A = 1 个 anchor；C = 4 个 anchor（需提前准备）
未验证的风险点
  • 4 段间 anchor 跳跃，转场是否突兀
  • 主角脸是否会随每段 anchor 切换微漂移
  • 4 张 anchor 制作成本（image-gen 模型成熟度）
来源：PDF 覆盖审计 + 60s 实战观察（R2V 段 3、4 场景执行不到位）。下一步：跑 1 个 60s 多 anchor 链式 demo，对比单 anchor 版本的场景执行度。

方案 9 · 5 秒一钩剧本结构（Hook-Every-5s）

待验证

Grok 60s Hack（309.9K 播放）的核心不是末帧串联，而是剧本侧的结构化约束：60s 切成 12 个 5s 钩子点，每点必须有动作变化 / 情绪反转 / 物体出现 / 光线变化之一。HH 单段 15s = 3 个钩子位，4 段刚好 12 个钩子覆盖完整长片。"AI 视频也可以有节奏"——观众第一次看到非平铺直叙的 AI 长片。

Scriptwriter Prompt 模板
任务：写一个 ${duration}s 故事
约束：1) 主角数 ≤ 2  2) 每 5s 一个 hook（hook 类型从 {动作 / 情绪 / 物体出现 / 光线变化} 选）
      3) 高张力开头，闭环结尾  4) 场景统一（不引入新地点除非剧本允许）
输出：JSON 数组，每段 5-8s，对应 visual_action / emotion / hook_type
       // 后端把 12 个 hook 自动映射成 4 段 × 15s 的 HH chain prompt
来源：Grok 60s Hack · 309.9K 播放。下一步：双对照 demo——「无 hook 约束」vs「12 hook 约束」生成同一 60s 故事，并排播放标注 hook 时间戳。

方案 10 · Synopsis → Storyboard → Visual Description 三段式自动分镜

待验证（P0 推荐）

Midjourney + Runway 课程（319.8K 播放）给出的工业级 pipeline 关键不在模型本身，而在 ChatGPT 三段式跑分镜：① 一句 logline → 16 句 synopsis；② synopsis → 16-scene storyboard JSON；③ 每 scene → 精细 visual description（喂给视频模型）。当前 HH 让用户直接写 prompt 跳过前两层，导致镜头随意。这是从「prompt 工程」升级到「分镜导演工作流」的关键。

三段式 API 设计
# POST /api/storyboard
input:  { logline, duration, style_tags }
step1:  LLM → synopsis （按时长切句）
step2:  LLM → storyboard JSON （每镜：景别 / 动作 / 情绪 / 光线 / 时长）
step3:  LLM → 五要素 prompt 数组 （直接喂 HH）
output: { synopsis, storyboard[], prompts[] }
        // 用户在 step2 可手动改 scene 顺序 / 添加镜头，然后一键批生成
价值（按汇报亮点）
  • 把 HH 从「单 prompt 抽奖」升级到「分镜导演工作流」
  • 用户可干预的环节多（step2 改 scene 顺序）
  • LLM 层包装，HH 不需要任何升级即可受益
实现难度
  • 纯 LLM 前置层，落地难度低
  • 需要前端拖拽重排 scene 卡片
  • 与 sol-1 R2V 链式 + sol-6 五要素 prompt 天然衔接
来源：Midjourney + Runway 完整课程 · 319.8K 播放。下一步：POC 三段式 API，用一句 logline 自动产出 60s storyboard + 4 段 R2V prompts。

小语种配音（音画分离工作流）

HH 只负责画面、配音交给专业 TTS + 对口型，同一画面 × N 种语言，一次画面投入多语言素材产出

致命
● 实测短板

阿拉伯语、印地语等小语种发音"AI 感强、不像真人"，无法直接投放至小语种地区。和 seedance 一样存在这个问题。

阿拉伯语：只是好了一些，但还是很奇怪；发音他们觉得很奇怪；都听出来是 AI 音，不像真人。
— 阿拉伯语运营评价
● 我们的应对：音画分离 + 本地化形象库

PDF 报告里阿拉伯语运营反馈"他们的模型不太"——这不只是配音问题，还包括中东 / 南亚人脸的形象真实度。所以方案是两层：(A) 音画分离解决"声音"；(B) 本地化形象库解决"脸"。

  1. Prompt 端：用「内心独白」模式（角色嘴唇全程闭合）避免 HH 自己尝试小语种发音
  2. 专业 TTS 出音：阿里云语音合成 / ElevenLabs 多语言 / 火山引擎 — 选有真人录音库的厂商
  3. 对口型（可选）：SadTalker / wav2lip / muscle-talk 等开源工具把配音对齐到视频口型
  4. 多语种快速产出：同一画面 × N 种语言配音，一次画面投入产出多语言素材
  5. 本地化形象库（新加）：沉淀中东男女 / 南亚男女各 5-10 张高质量 anchor 参考图，R2V 调用时用 character1 锁定本地化形象，避免 HH 默认偏白人 / 东亚的训练倾向。覆盖 PDF 的 P3 / P4 痛点
项目里已实现
万花筒预设 · 真人短剧规则中已包含「内心独白处理」段："角色嘴唇全程闭合，音频层保留独白声音，画面层禁止任何张嘴动作"（Card2 万花筒）。
关键调用骨架（TTS → 合并 → 对口型）
# 1. 阿里云 NLS 生成阿拉伯语配音
curl -X POST "nls-gateway.aliyuncs.com/stream/v1/tts" -d '{"voice":"yousuf","language":"ar-SA",...}'

# 2. ffmpeg 合并无声画面 + 配音
ffmpeg -i hh-mute.mp4 -i ar.mp3 -c:v copy -c:a aac -shortest out.mp4

# 3. （可选）wav2lip 对口型
python inference.py --face hh-mute.mp4 --audio ar.mp3 --outfile lipsync.mp4

PROBLEM & SOLUTION

循环动画（首尾闭合）

三档分级方案：简单几何 → Lottie，写实材质 → HH+裁剪，过渡形态 → HH+prompt

严重
● 实测短板

头像框、勋章光效等需要首尾无缝循环的场景，HH 不支持，prompt 约束失败。

无法做循环动画（prompt 上约束失败，不支持首尾），导致这一块无法派上用场。
— 动效侧测试结论
● 我们的应对：三档分级方案

按需求复杂度选不同方案，按场景分工。

  1. A 档（HH + prompt 强约束）：prompt 明确"起始姿态 = 结束姿态" + "全程匀速循环"——简单缓动可用，复杂动作大概率失败
  2. B 档（HH + 后期裁剪）：跑 15s，用 FFmpeg 自动找动作相似度最高的两帧作循环起止点，导出片段后做 crossfade 拼接
  3. C 档（Lottie / GSAP 矢量动画）：勋章光效、头像框旋转等几何确定的元素——直接用前端矢量动画，文件 KB 级、无限循环、可换色
  4. 决策矩阵：简单几何 → C；写实材质 → B；过渡形态 → A
  5. 优先级提示（针对 PDF 痛点）：头像框 / 勋章 / loading 等 UI 类循环 = C 档优先（Lottie/GSAP）。PDF 报告里明确"无法派上用场"的场景就是这类几何确定的元素，不要让设计师误选 A 档浪费 token
推荐工具组合
B 档实施：FFmpeg scenedetect + ssim 相似度算法自动找循环点。
C 档实施：Lottie / Rive / GSAP 三选一，设计师 AE 出 Lottie，前端直接 npm 引入。
B 档：FFmpeg 找循环点 + 加 crossfade
# SSIM 相似度找循环起止
scenedetect -i loop-source.mp4 detect-content list-scenes -o report.csv

# 截目标段 + crossfade 形成无缝循环
ffmpeg -i source.mp4 -ss 2.0 -t 3.5 -filter_complex "fade=in:0:15,fade=out:75:15" loop.mp4
C 档：Lottie 矢量勋章（文件 ~10KB）
import lottie from 'lottie-web';
lottie.loadAnimation({ container: '#medal', loop: true, path: '/medals/gold-glow.json' });

礼物：画面质量没大问题，但动画节奏、逻辑、运镜对比，seedance2.0 依旧比 HappyHorse 更优秀。

  if (tags.includes('stylized') || tags.includes('mecha')) return 'seedance-2.0'; // 节奏更强
  return 'happyhorse-1.0';                                              // 真人/电影/电商默认 HH
}

CAPABILITY EXTENSION

多主角对手戏（character1 + character2）

HH 1.0 R2V 原生支持 1-9 张参考图，prompt 用 character1 / character2 引用——一次出双人对话/互动场景，买量侧两人种草、母女、人+宠物高频可复用

扩展
● HH 1.0 原生能力

R2V 模式的 reference_urls 上限 9 张，prompt 中用 character1 指第 1 张、character2 指第 2 张……按上传顺序对应。我们项目里已支持 1-9 张参考图字段。典型买量场景：两人对话种草、母女对手戏、人+宠物互动、双 IP 联动。

调用骨架
media: {
  reference_urls: ["oss://.../character1.jpg", "oss://.../character2.jpg"],
  prompt: "character1 与 character2 隔桌而坐，character1 举起咖啡杯，character2 用手势讲述故事..."
}
A · 对照（单 character R2V 15s）
仅传入主角 character1（韩系女生 anchor），prompt 描述独角戏场景。结果预期：主体一致性好，但单人画面叙事单薄
B · 升级（双 character R2V 15s）
传入 character1 + character2（亚洲男生），prompt 用语法 character1 / character2 描述两人互动。结果预期：双人对手戏成功生成，主体身份与互动节奏并存
06
CAPABILITY EXTENSION

五要素 prompt 结构化（Cinematic Prompt Recipe）

基于 Dan Kieft 拆解：Camera + Subject + Action + Environment + Lighting 五要素结构化书写 → 同一题材出片质量大幅跃升，避免"自由 prompt 抽奖"

扩展
● 应对：把 prompt 拆成 5 个槽位

Dan Kieft 在「Why My AI Videos look Ultra Realistic」教程中提出：电影感来自 lighting / composition / depth / color / emotion 五个层次。落到 prompt 工程上就是固定 5 个槽位，每次写 prompt 套用模板，杜绝"想到啥写啥"。

5 要素 Prompt 模板
[Camera]     中景 / 35mm 镜头 / 缓慢推进 dolly-in
[Subject]    25 岁韩系女生，长发披肩，米白针织衫，3/4 侧脸
[Action]     双手捧起白瓷咖啡杯，杯沿轻触嘴唇，蒸汽袅袅升起
[Environment] 北欧原木风咖啡馆，窗边木桌，桌上一本翻开的书
[Lighting]   柔和午后金色窗光 45 度斜入，胶片质感
A · 对照（自由 prompt）
Prompt = "一个女生在窗边喝咖啡"。结果预期：HH 自由发挥，光线、构图、运镜随机；可能出片可用但电影感弱
B · 升级（5 要素结构化）
Prompt = [Camera]+[Subject]+[Action]+[Environment]+[Lighting] 5 段填空。结果预期：光线方向精确、构图稳定、镜头运动可控，明显的"电影感"提升

CAPABILITY EXTENSION

运镜预设库（Camera Movement Preset）

Joshua Mayo 教程整理出 25-30 种工业运镜（Dolly Zoom / Orbit / Crash Zoom...）→ 写成预设模板挂进万花筒，用户下拉就用，避免每次手写 camera 描述

扩展
● 应对：把工业运镜写成预设

Higgsfield 网站把运镜做成一键 preset，我们用 prompt 模板实现等价能力。每个 preset = 一段 [Camera Movement: ...] 头部 + 主体场景描述，挂进万花筒分类，用户选完直接用。

A · 对照（无 camera 指令）
仅描述主体与场景，不写 camera。结果预期：运镜随机、节奏平淡
B · 升级（3 种代表性预设）
B1 · Dolly Zoom 眩晕
B2 · Orbit 环绕
B3 · Crash Zoom 推近
3 种工业运镜各 5s：Dolly Zoom（Hitchcock 眩晕）/ Orbit（180 度环绕）/ Crash Zoom（暴力推近）。下一步把 25+ 种 preset 整理进万花筒
08
CAPABILITY EXTENSION

服装多版本批量产出（Wardrobe Iteration）

同一主角 anchor + prompt 切换穿搭风格 → 一个 IP 产出 N 套素材，买量侧"同一代言人 × N 个穿搭场景"批量化产能

扩展
● 应对：固定 anchor，prompt 切风格

R2V 锚定主角面孔，prompt 用 <服装风格> 槽位切换。同一 IP 可以一次产出休闲/正装/运动/居家等 N 套素材，避免每次重新生成主角面孔。

调用骨架
reference_urls: [anchor_oss],
prompt: "character1 换装成<休闲风格>：浅蓝色丹宁牛仔外套 + 内搭白色 T 恤...保持人物面孔与气质一致。"
// 同一 anchor，切换 prompt 中的 <风格> → 一键产出 N 套
A · 原服装
B1 · 休闲牛仔
B2 · 知性正装
B3 · 街头运动

关键观察：4 个版本主角面孔保持一致（R2V 锚定的力量），但服装、配饰、氛围按 prompt 切换。对买量侧的意义：同一主角一次拍摄成本可衍生 N 套素材，A/B 测试速度大幅提升。
